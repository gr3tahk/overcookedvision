Say: "We're going to walk through a research project that asks a deceptively simple question — if you give an AI a picture of a kitchen, can it cook? The answer turns out to be: mostly no, but in interesting ways."

MOTIVATION

Why Overcooked?

Overcooked looks simple. It forces agents to coordinate, track state, and execute a precise sequence of actions.

The game requires:

  • Divide work — neither agent can complete the task alone
  • Multi-step planning — soup takes 3 onions + dish + delivery
  • Precise positioning — must face station correctly to interact
  • Error recovery — wrong action can block later progress
  • Partner tracking — who is doing what, right now?

THE TASK (CRAMPED ROOM)

Layout: XXPXX / O O / X X / XDXSX
Stations: Onion (×2), Pot, Dish, Serve

Recipe: 3 onions → pot → dish → serve
Budget: 60 ticks · 3 trials
Text representations may bottleneck partner tracking in dynamic layouts. Visual grounding might help — or hurt.

Say: "The Overcooked game looks like a toy problem but every decision has a spatial and temporal dependency. You have to be in the right place, facing the right direction, at the right time, while your partner is doing something complementary. That's exactly the kind of situation where we'd expect vision to help — or to reveal its limits."

BACKGROUND

The Original: Overcooked-AI

Carroll et al. (2019) built Overcooked-AI as a research environment for studying human-AI coordination under layout constraints.

  • Standard RL benchmark — agents trained via self-play
  • 5 canonical layouts (Cramped Room, Asym. Adv., Coord. Ring, Forced Coord., Counter Circuit)
  • Spatial reasoning is central — positions, facings, and object states all matter
  • Provides strong RL baselines for comparison
Carroll et al. built it for human-AI coordination research — not for LLMs. The existing LLM benchmark (Agashe et al.) was too tightly coupled to GPT-4 API to reuse, so we rebuilt the evaluation framework from scratch.

5 LAYOUTS

AA Asymmetric Advantages · role split
Ring Coordination Ring · partner tracking
FC Forced Coordination · counter-passing
CC Counter Circuit · complex routing

WHY WE REBUILT IT

Agashe et al.'s LLM-Coordination benchmark was tightly coupled to GPT-4 API, making open-source/local model comparison impossible without rebuilding scaffolding from scratch.

Say: "The RL agents learn from millions of steps. We're asking LLMs and VLMs to do the same thing with zero training — just a prompt and a game state. The RL baselines give us a ceiling to look up at."

PRIOR WORK

Collab-Overcooked Inspired Us

Agashe et al. (2025) scaffolded LLMs with a Memory · Reasoning · Grounding architecture and evaluated GPT-4-turbo, GPT-4o, GPT-3.5-turbo across all five layouts.

GPT-4 MATCHED RL

In environment-focused layouts, GPT-4-turbo matched or exceeded RL baselines — a strong result for text-only coordination.

THEORY OF MIND BROKE

When reasoning about a partner's goals was required, even GPT-4-turbo broke down. Multi-agent coordination remains hard.

PROCESS METRICS

Introduced SR, TES, PC — process-oriented evaluation. A model can fail the task but still make meaningful progress.

Key metrics they introduced (which we reuse):

SR — Success Rate (binary)
TES — Task Execution Score (LCS vs reference)
PC — Progress Completeness (per-agent)
The biggest inspiration: process-oriented evaluation. A model that almost delivers the soup is very different from one that does nothing.

Say: "The collab overcooked work taught us two things. First, the Memory-Reasoning-Grounding scaffold was so custom that you couldn't cleanly separate model capability from scaffolding quality. Second — and this is the part that inspired us — their process metrics are much richer than just success/fail."

OUR WORK

Our Project: Vision Overcooked

Research question: Do VLMs outperform text-only LLMs when agents must coordinate spatially? Does seeing the kitchen help them cook?
  • Use multimodal models on rendered game screenshots
  • Compare against text-only baselines under identical scaffolding
  • Keep full traces so every failure can be inspected tick by tick
  • Reuse process metrics (TES, PC) from Collab-Overcooked
The only variable between LLM and VLM conditions is input modality — text state description vs. rendered screenshot. Everything else is identical.

CHALLENGE

Why We Rebuilt the Benchmark

The existing Collab-Overcooked setup was not clean enough for controlled visual experiments.
  • Tightly coupled to GPT-4 API — local/open-source replication impossible without rewriting everything
  • Scaffolding conflated with model — the Memory/Reasoning/Grounding stack was so custom that results couldn't be attributed to model capability alone
  • No controlled VLM path — no prior benchmark directly compared VLM and LLM agents under identical scaffolding in multiplayer
  • Partitioned layouts broken — Asymm. Adv. and Forced Coord. needed counter-passing logic that wasn't there
This became a bigger part of the project than expected. Rebuilding reduced noise and gave us full control, but limited how many experiments we could run.

WHAT WE BUILT

  • BFS pathfinding grounding layer
  • Real-time trace logging (tick-by-tick)
  • Identical scaffolding for all models
  • TES / PC metric implementation

Say: "We planned to run the existing benchmark and plug in visual inputs. Instead we ended up rebuilding it from scratch. That was the right call scientifically — but it meant we had less time for experiments. This tradeoff shows up in the results."

HIGH-LEVEL GOAL VOCABULARY

pickup_onion
place_onion_in_pot
pickup_dish
pickup_soup
deliver_soup
wait

LOW-LEVEL NAVIGATION

BFS pathfinder translates each goal into primitive NORTH/SOUTH/EAST/WEST/INTERACT actions. Model never has to plan movement directly.

KEY DESIGN CHOICES

  • Rule-based logic handles obvious inventory decisions — model only queried for genuine coordination decisions
  • Both agents act simultaneously each tick
  • Full trace saved: position, orientation, held object, pot state, every tick

TRACE OUTPUT

Every run saves tick-by-tick game state enabling post-hoc replay, metric computation, and qualitative failure analysis.

Say: "The two-level architecture is key. The model doesn't have to think about which direction to walk — BFS handles that. The model only has to answer: given the current state, what is the right high-level goal? That isolates the planning problem from the navigation problem."

METRICS

How We Scored the Agents

SR — SUCCESS RATE

soups_delivered > 0

Binary: did the agents deliver at least one soup within 60 ticks? Primary outcome — but tells us nothing about how they behaved.

GPT-5.4 LLM: 3/3 ✓   Qwen3-8B: 0/3

TES — TASK EXECUTION SCORE

LCS(actual, reference) / max(|actual|, |ref|)

Longest-common-subsequence alignment with the optimal reference plan. High TES with SR=0 means on track but ran out of time.

GPT-5.4 LLM: 0.859   Qwen3-8B: 0.000

PC — PROGRESS COMPLETENESS

avg( TES_agent0 , TES_agent1 )

Per-agent progress averaged. Reveals asymmetry: one agent working, other idle shows up here. Low PC = both agents failed to contribute.

Qwen3-4B: PC=0.25 (agent 0 = 0.00)

INVALID ACTIONS

outputs ∉ action vocabulary

Outputs the model produced that couldn't be parsed or executed. A different failure mode — interface compliance, not planning.

Qwen3-8B: 29 invalids → TES = 0.000
Reference plan: pickup_onion → place_in_pot (×3) → pickup_dish → pickup_soup → deliver_soup

Say: "Success is binary but the world isn't. TES and PC let us distinguish the model that got 80% of the way there from the model that did nothing. And invalid action count is a completely different signal — it means the model couldn't even produce legal outputs."

FINDING 1

Finding 1: Scale Mattered Most

Only the strongest model completed the task reliably. The capability gap between GPT-5.4 and everything else was visible even on this simple layout.
  • GPT-5.4 LLM: 3/3 — the only model to complete every trial
  • GPT-5.2 LLM: 1/3 — same architecture, weaker planning
  • All local models: 0/3 — none completed regardless of size or modality
  • Scale gap > modality gap — GPT-5.4 vs 5.2 is a bigger gap than LLM vs VLM within the same model family
This matches Collab-Overcooked's scaling pattern: smaller models struggle even before task complexity increases.

FINDING 2

Finding 2: Vision Was Not Magic

At both GPT-5.2 and GPT-5.4, the VLM variant underperformed its LLM counterpart. Seeing the kitchen didn't guarantee acting well in it.
  • Vision runs often knew the recipe — they picked up onions, they knew soup needed 3 — but couldn't execute precisely
  • Failures at positioning — model had to know where to stand, which way to face, and when an interaction is legal
  • Visual grounding added difficulty — the image had to be converted into a grid coordinate and then an action
  • Aligns with BALROG (Paglieri et al. 2025) — VLMs perform worse than text-only LLMs on game tasks

TRACE ANALYSIS

Trace: What Failure Looked Like

GPT-5.2 LLM trials 1+2 — partial progress, then timeout. Good semantic plan, unreliable low-level control.
Common pattern: dish → go to pot → interact → no-op loop → timeout. Model said it was picking up soup, but state didn't change.
Key insight: Good semantic plan ≠ reliable low-level control. Grounding failures, not recipe confusion, caused most timeouts.

Say: "Watch trial 2 — the agents get two onions in the pot, then stall. They have the right idea. But they can't execute the next step cleanly. The model output the right action name, but something about the state didn't match what the model expected. This is the grounding gap."

CONTEXT

Connection to Collab-Overcooked

This comparison is conceptual, not direct. Different models, different scaffolding, different task scope. We were inspired by the evaluation philosophy — not running a controlled ablation against their results.

Collab-Overcooked (Agashe et al.)

  • Language-only agents
  • Memory / Reasoning / Grounding scaffold
  • GPT-4-turbo, GPT-4o, Mixtral
  • 5 layouts, many tasks
  • TES / ITES / PC metrics
  • Collaboration initiation / response

Vision Overcooked (Ours)

  • Multimodal visual input
  • Minimal, identical scaffolding
  • GPT-5.x + Qwen3 local models
  • 1 layout, 1 task (scope tradeoff)
  • Same SR / TES / PC metrics
  • Focus: grounding plans into actions
The key methodological contribution: identical scaffolding across all conditions isolates modality as the variable. Collab-Overcooked's scaffold differences made cross-model comparison harder.

Say: "When we say our results differ from theirs, we're not making a controlled comparison. Different models, different scaffolding. What we borrowed was the evaluation philosophy — measure process, not just outcomes — and the metric definitions."

LIMITATIONS

Limitations + Future Work

WHAT HELD US BACK

  • Rebuilding took time — benchmark construction ate time budgeted for experiments. We ran fewer conditions than planned.
  • One layout, one task — Cramped Room is the easiest layout. The LLM > VLM finding may not hold on spatially harder layouts like Coordination Ring where partner tracking matters more.
  • Visual prompt design rough — VLM prompting was not tuned. Better visual representations (annotated screenshots, highlighted agents) might close the gap.
  • Local compute limits — Qwen models ran on BU SCC under resource constraints

WHAT COMES NEXT

  • More layouts — Coordination Ring is where vision might actually help. Test the 5 canonical layouts with current setup.
  • Better visualizations — annotated screenshots, agent position highlights, bounding boxes on objects
  • Automatic error classification — parse failure vs. planning failure vs. grounding failure are different; automate their detection
  • More model families — Claude, Gemini, Llama-VL under same controlled scaffolding
The benchmark is a proof of concept. It needs more task variety and better visualization experiments before we can make strong claims about modality.

Say: "The honest limitation is that we tested one layout. Cramped Room is almost too easy — it probably doesn't require vision. The interesting test would be Coordination Ring, where the ring shape means your partner is always moving around and you need to track them. That's where vision might actually win."

CONCLUSION

Models can often
see the kitchen.
Most still cannot
reliably cook.

  • High-level understanding is not enough — recognizing what to do ≠ executing it correctly
  • Grounded, step-by-step execution is the bottleneck — the hard part is turning visual state into a stable sequence of correct actions
  • Process traces made failures visible — TES and PC revealed that most failed models were partially right, not totally wrong
The project shows that visual planning is not just about recognizing the state.

The hard part is turning that state into a stable sequence of correct actions.

That gap — between perception and grounded execution — is where future work should focus.
